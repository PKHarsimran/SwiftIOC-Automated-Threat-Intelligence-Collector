Operational playbook

A lightweight, repeatable workflow that helps responders keep SwiftIOC data fresh, actionable, and measurable inside their existing tooling.

  1. 1

    Validate freshness and scope

    Start each review by confirming the collection window and overlaps so downstream teams know exactly how recent and comprehensive the data is.

    • Compare timeline metrics with your detection coverage SLAs.
    • Alert stakeholders if the multi-source overlap count drops unexpectedly.
    • Bookmark the diagnostics summary for quick health checks during incidents.
  2. 2

    Promote the right exports

    Select the format that best aligns with your automation stack and make it part of a scheduled sync so analysts always see the latest indicators.

    • Use the JSON feed for orchestration platforms and enrichment microservices.
    • Stream the JSONL file into data lakes to preserve historical context.
    • Create a GitHub Action that publishes CSV/TSV artifacts for SIEM uploads.
  3. 3

    Close the loop with feedback

    Track which sources and indicator types deliver the most value so you can iterate on the playbook and retire noisy feeds.

    • Map source totals to your internal asset or client groups.
    • Collect analyst feedback directly inside pull requests for new source onboarding.
    • Document high-confidence tags and share them with your threat hunting community.

Recommendations for GitHub Pages deployments

Keep the published dashboard trustworthy and easy to evangelise with these lightweight improvements tailored for GitHub Pages hosting.

Automate publishing

Use a scheduled GitHub Action that runs the SwiftIOC collector, commits the refreshed public/ artefacts, and triggers a Pages deploy so the site never goes stale.

  • Store API secrets as GitHub Action secrets with least-privilege scopes.
  • Attach a job summary that links directly back to the latest dashboard build.

Deliver rich previews

Add Open Graph images to public/assets/ and reference them via metadata so sharing the dashboard in Slack or Teams displays an informative preview.

  • Generate the image as part of the pipeline to highlight total indicators and freshness.
  • Optimise the asset below 512 KB to keep GitHub Pages fast.

Instrument engagement

Track which download options are most popular by wiring privacy-friendly analytics such as Plausible or Azure Application Insights through a consent-aware toggle.

  • Sample page views per environment instead of collecting user-level data.
  • Expose opt-in messaging so defenders understand the measurement scope.

Boost discoverability

Publish a security.txt and sitemap.xml alongside the dashboard to help external teams and search engines locate support contacts and machine-readable content.

  • Reference the security contact in your organisation profile and repository README.
  • Mirror the sitemap generation inside the SwiftIOC pipeline for zero-maintenance updates.

Frequently asked questions

Share these quick answers with stakeholders rolling out the GitHub Pages dashboard.

How often should the dashboard refresh?

Align the schedule with your highest-priority detection workflows. Many teams rebuild the site hourly during active incidents and at least daily during steady state operations.

Can we restrict access to the published data?

GitHub Pages is public by design. If you require access controls, mirror the generated public/ directory to a private S3 bucket or internal static site hosting service and reuse these assets there.

Where should contributors report issues?

Encourage teams to open GitHub issues in the repository so feedback stays close to the infrastructure-as-code. The diagnostics report helps triage whether the problem stems from data quality or hosting.
